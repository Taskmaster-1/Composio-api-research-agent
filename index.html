The patterns

the headline, not the table

74 of 100 apps could become agent toolkits this week — the gate is business, not technology.

Where an app isn't buildable, the blocker is almost never a missing API — it's who is allowed to hold credentials. Top blockers found: paid plan required (12), app review or admin gate (11), other (6).

OAuth2 dominates (56/100)

OAuth2 appears on 56 apps and API keys on 54 — many apps ship both: OAuth for user-delegated access, keys for server-to-server. A toolkit platform must treat dual-auth as the default case, not the edge case.

Self-serve is the norm — gating is categorical

Communications is the most open category (100% self-serve); Finance & Fintech is the most gated (30%). Openness clusters by who the buyer is: developer-first products open the door, enterprise- and partner-sold products keep a human in the loop.

4 easy wins with no MCP yet

Apps that are buildable, self-serve, and have no MCP server found — greenfield for Composio. The gated list (5 apps) is the outreach queue, not a dead end.

The agent

python · groq/openrouter/gemini fallback chain · serper search
01

Search ×3

Per app: docs & auth, MCP servers, credential access — via Google (Serper).

02

Read the docs

Downloads the top docs pages and reads the text — snippets alone lie.

03

Extract

One LLM call → strict JSON: auth, access, API, MCP, verdict, evidence URLs, confidence. Guessing forbidden.

04

Checkpoint

results.json written after every app — crash-safe, resumable.

05

Verify → fix → verify

A deterministic checker (no LLM) re-fetches all evidence URLs and tests every claim; flagged records get corrected and re-checked.

Where a human was needed

Defining the schema and what “self-serve” means. Unblocking run #1, which died on Gemini free-tier 429s (kept in the repo as artifacts_run1_gemini_429.json) — replaced with a Groq → OpenRouter → Gemini fallback chain. When free-tier quotas kept choking, 16 apps were finished by a human-driven research pass (same schema, same evidence rules — marked researched_by in the data). Hand spot-checking 13 claims against real docs and correcting 9 records. The agent did everything else.

What the agent got wrong (kept honestly)

The hand check caught real failure modes: entity confusion (Fathom's MCP evidence pointed at Fathom Analytics, a different company), context bleed (Mermaid CLI got "API Key" auth from unrelated search results — it has no auth at all), over-optimism on gated APIs (PitchBook needs a standalone contract, not "partial"), and over-caution (Zendesk's free trial has full API access). 1 records self-report low confidence. The spot-check table below shows every hit and miss.

04

How we know it's right

accuracy is what matters most

Machine loop — deterministic, no LLM

The checker re-fetches every record's evidence URLs and mechanically tests each claim (URL liveness + keyword support). It cannot hallucinate agreement — a dead evidence link can never count as support. Pass 1: 398/489 claims supported (81%), 25 dead evidence links. After the fix pass (evidence-URL repairs + 4 claim corrections), pass 2: 399/488 (82%), dead links down to 21. Most remaining flags are docs the static fetcher can't read (JS-rendered or bot-blocked: Salesforce, Meta, Datadog) — a stated limitation, covered by the human loop, not papered over by loosening the checker.

Human loop — where judgment lives

13 field-level spot checks against the real docs by hand — the agent had 8/13 right (62%) before fixes; every miss was corrected with a cited docs page and re-verified, and the machine loop's flags triggered 9 record corrections in total. Sample weighted toward low-confidence and gated verdicts, because that's where agents bluff.
